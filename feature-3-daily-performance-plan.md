# خطة تنفيذ فيتشر الأداء اليومي

تاريخ الوثيقة: 2026-03-29  
نطاق الوثيقة: تحليل + خطة تنفيذ كاملة لميزة الأداء اليومي للموظف وتقييمها من الإدارة/المقيم

## 1) ملخص المطلوب (حسب وصفك)

1. صفحة للموظف باسم "الأداء اليومي":
- الموظف يسجل يوميا ما أنجزه.
- الحقول المطلوبة: اسم المشروع، وصف ما تم، رفع صورة أو أكثر، أو ملف اختياري.
- عرض يومي واضح مع إمكانية الرجوع لليوم السابق أو الأسبوع السابق.

2. صفحة للمقيمين (admin/manager/hr/user) باسم "تقييم الأداء اليومي":
- عرض الموظفين على شكل كروت (اسم + صورة).
- لكل موظف: هل سجل اليوم أم لم يسجل.
- عند وجود سجل: يظهر اسم المشروع والوصف والمرفقات مع فتح الصور/الملفات.
- تحت الكارت: تقييم نجوم من 1 إلى 5 + تعليق.

3. فلترة وتقارير في صفحة التقييم:
- فلتر بالتاريخ.
- فلتر بموظف محدد.
- فلتر حسب حالة التسجيل (سجل/لم يسجل).
- إحصائية أعلى الصفحة: عدد من سجلوا اليوم.

4. بعد تقييم السجل:
- الموظف يرى التقييم والتعليق الخاصين بسجله.

---

## 2) تحليل المشروع الحالي وربط الميزة بالواقع

تمت مراجعة البنية الحالية للتأكد من التكامل وعدم كسر ما هو موجود:

1. نظام الصلاحيات الحالي يعتمد على middleware role ويستخدم الأدوار:
- admin
- manager
- hr
- employee
- user

2. المسارات الحالية منظمة داخل مجموعات role واضحة.

3. هناك نمط قائم قريب جدا من المطلوب:
- صفحة موظف يرى تقييمه (مهامي).
- صفحة مقيم يقيّم عناصر (تقييم المهام).
- رفع ملفات موجود في حسابي (avatar) ويستخدم disk public.

4. الواجهة الحالية تعتمد:
- Blade + Tailwind
- Sidebar ديناميكي حسب الدور
- كروت وتصميم عربي جاهز ويمكن البناء عليه مباشرة.

5. توجد علاقة User -> Employee مفعلة بالفعل، وهذا يسهل تحديد صاحب السجل اليومي بدقة.

الخلاصة: الميزة يمكن إضافتها بشكل نظيف كـ Module جديد دون إعادة هيكلة النظام.

## 3) تصميم قاعدة البيانات المقترح

لتحقيق متطلباتك مع مرونة مستقبلية، الأفضل تقسيم البيانات إلى 3 جداول.

### 3.1 جدول السجل اليومي

الاسم المقترح: daily_performance_entries

الأعمدة:
- id
- employee_id (FK -> employees)
- work_date (date)  // تاريخ الأداء
- project_name (string 255)
- work_description (text)
- submitted_at (timestamp nullable)
- created_at
- updated_at

قيود مهمة:
- unique(employee_id, work_date)  // سجل واحد لكل موظف لكل يوم

فهارس:
- index(work_date)
- index(employee_id)

### 3.2 جدول المرفقات

الاسم المقترح: daily_performance_attachments

الأعمدة:
- id
- entry_id (FK -> daily_performance_entries)
- disk (string 50) default public
- path (string 500)
- original_name (string 255)
- mime_type (string 150 nullable)
- file_size (unsignedBigInteger nullable)
- is_image (boolean default false)
- created_at
- updated_at

فهارس:
- index(entry_id)
- index(is_image)

### 3.3 جدول التقييمات

الاسم المقترح: daily_performance_reviews

الأعمدة:
- id
- entry_id (FK -> daily_performance_entries)
- reviewer_user_id (FK -> users)
- rating (unsignedTinyInteger)  // 1..5
- comment (text nullable)
- reviewed_at (timestamp nullable)
- created_at
- updated_at

قيود:
- unique(entry_id, reviewer_user_id)  // كل مقيم يقيّم السجل مرة واحدة ويقدر يعدلها

فهارس:
- index(entry_id)
- index(reviewer_user_id)
- index(rating)

## 4) قواعد العمل (Business Rules)

1. الموظف role=employee فقط يضيف/يعدل سجله اليومي.

2. التسجيل اليومي يكون على work_date:
- افتراضيا تاريخ اليوم.
- مسموح تعديل سجل نفس اليوم.
- خيار الرجوع/استعراض الأيام السابقة فقط (والتعديل يعتمد قرار إداري).

3. المرفقات:
- متعددة (images + files).
- حد أقصى عدد ملفات لكل سجل (مقترح 5).
- حد حجم الملف الواحد (مقترح 10MB).
- الامتدادات المسموحة:
	- الصور: jpg,jpeg,png,webp
	- الملفات: pdf,doc,docx,xls,xlsx,txt,zip

4. التقييم:
- الأدوار المسموح لها: admin, manager, hr, user.
- التقييم من 1 إلى 5.
- التعليق اختياري.
- يسمح بالتعديل من نفس المقيم (upsert).

5. عرض حالة الموظفين يوميا:
- سجل = يوجد entry لليوم.
- لم يسجل = لا يوجد entry.

6. ما يظهر للموظف بعد التقييم:
- متوسط التقييم (اختياري لكنه مفيد).
- آخر التقييمات مع اسم المقيم والتعليق.

## 5) تصميم الصفحات المطلوبة

### 5.1 صفحة الموظف: الأداء اليومي

المكونات:
- Header يومي مع التاريخ الحالي + أزرار: اليوم السابق / التالي / هذا الأسبوع.
- فورم:
	- اسم المشروع
	- وصف ما تم إنجازه
	- رفع متعدد للمرفقات
- Card "تقييمات اليوم" يظهر فيها:
	- متوسط النجوم
	- قائمة التعليقات

تجربة الاستخدام:
- إذا سجل اليوم موجود: الفورم يعمل كـ تعديل وليس إضافة جديدة.
- إذا لا يوجد سجل: الفورم إضافة جديدة.
- تحت الفورم: Timeline صغير لآخر 7 أيام (حالة سجل/لم يسجل).

### 5.2 صفحة المقيم: تقييم الأداء اليومي

المكونات:
- شريط فلاتر:
	- التاريخ
	- الموظف
	- الحالة (all/submitted/not_submitted)
- بطاقات إحصائية أعلى الصفحة:
	- إجمالي الموظفين
	- عدد من سجلوا
	- عدد من لم يسجلوا
	- نسبة التسجيل
- Grid كروت الموظفين:
	- صورة + اسم
	- Badge حالة التسجيل
	- عند وجود سجل: المشروع + الوصف + المرفقات
	- نموذج تقييم نجوم 1..5 + تعليق

سلوك الفلاتر:
- افتراضيا يعرض تاريخ اليوم.
- يتيح استعراض أي يوم سابق بسهولة.

## 6) الربط الفني داخل المشروع الحالي (Integration Points)

### 6.1 Routes

داخل مجموعة auth + force_password_change الحالية:

1. مجموعة employee:
- GET صفحة الأداء اليومي
- POST حفظ/تحديث سجل اليوم
- DELETE حذف مرفق (اختياري)

2. مجموعة admin/manager/hr/user:
- GET صفحة تقييم الأداء اليومي
- POST/PUT حفظ تقييم السجل (upsert)

### 6.2 Sidebar

إضافة روابط جديدة حسب الدور:
- للموظف: "الأداء اليومي"
- للمقيمين: "تقييم الأداء اليومي"

### 6.3 Models Relations

إضافة علاقات جديدة في:
- User (التقييمات التي كتبها)
- Employee (سجلات الأداء اليومية)

### 6.4 Controllers + Services

مقترح تقسيم نظيف:
- Controller للموظف (عرض/حفظ السجل)
- Controller للتقييم (عرض الكروت + حفظ التقييم)
- Service لمعالجة السجل والمرفقات
- Service لمعالجة التقييم والفلاتر والإحصائيات

### 6.5 Validation

يفضل Form Request classes منفصلة:
- Store/Update Daily Performance Entry Request
- Upsert Daily Performance Review Request

## 7) الاستعلامات الحرجة والأداء

لتجنب بطء صفحة التقييم اليومية:

1. تحميل الموظفين مع profile/avatar مرة واحدة.

2. تحميل entries لليوم المحدد بـ eager loading:
- entry.attachments
- entry.reviews.reviewer

3. استخدام whereDate(work_date, selectedDate) مع index على work_date.

4. حساب الإحصائيات على مستوى Query وليس loop في الواجهة.

5. Pagination للكروت عند زيادة عدد الموظفين (مقترح 12 أو 20 لكل صفحة).

---

## 8) الأمان والصلاحيات

1. منع أي role غير مصرح له من الوصول للمسارات.

2. الموظف لا يستطيع:
- رؤية سجلات موظفين آخرين.
- تعديل تقييمات.

3. المقيم لا يستطيع:
- تعديل سجل الموظف نفسه.
- حذف مرفقاته.

4. حماية رفع الملفات:
- تحقق صارم من MIME + extension + الحجم.
- تخزين على disk public مع path منظم مثل:
	- daily-performance/{employee_id}/{work_date}/...

5. عند عرض الملف:
- استخدام رابط آمن داخلي (أو Storage URL مع ضبط صحيح).

## 9) اختبارات مطلوبة لتفادي الأخطاء

### 9.1 Feature Tests

1. employee يقدر ينشئ سجل يومي.
2. employee يقدر يعدل سجل نفس اليوم.
3. منع إنشاء سجلين لنفس الموظف ونفس التاريخ.
4. رفع مرفقات متعدد يعمل بنجاح.
5. رفض ملف بحجم/امتداد غير مسموح.
6. admin/manager/hr/user يقدروا يشوفوا صفحة التقييم.
7. المقيم يقدر يحفظ تقييم 1..5.
8. نفس المقيم يعدل تقييمه (upsert).
9. employee يرى التقييمات بعد حفظها.
10. فلتر سجل/لم يسجل يعمل صحيح.

### 9.2 Authorization Tests

1. user غير مصرح له لا يدخل صفحة الموظف.
2. employee لا يدخل صفحة تقييم الأداء.
3. guest لا يدخل أي صفحة محمية.

## 12) نقاط قرار قبل البدء (أسئلة مهمة)

1. هل تريد السماح للموظف بتعديل سجل أي يوم سابق؟
- خيار A: اليوم فقط
- خيار B: آخر 3 أيام
- خيار C: أي يوم سابق

2. التقييم الذي يظهر للموظف يكون:
- كل تقييمات المقيمين + تعليقاتهم
- أم متوسط النجوم + أحدث تعليق فقط

3. هل التعليق في التقييم إجباري عند النجوم الأقل من 3؟ (مفيد للجودة)

4. حد المرفقات:
- كم ملف لكل سجل؟
- هل نسمح بملفات zip أم نكتفي بالمستندات والصور؟

5. هل role=user هو المقيم الرئيسي فقط، أم admin/manager/hr أيضا يقيّمون فعليا؟
- حاليا الخطة تدعم الجميع كما طلبت.

## 14) تعريف النجاح (Acceptance Criteria)

1. الموظف يستطيع تسجيل أدائه اليومي مع مرفقات متعددة بدون أخطاء.
2. المقيم يرى الكروت اليومية مع حالة سجل/لم يسجل.
3. المقيم يضيف تقييم 1..5 وتعليق.
4. الموظف يرى التقييم والتعليق على سجله.
5. الفلاتر والإحصائيات اليومية تعمل بدقة.
6. الصلاحيات تمنع أي وصول غير مصرح.
7. جميع السيناريوهات الأساسية مغطاة باختبارات Feature.
