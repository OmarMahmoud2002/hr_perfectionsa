# تحليل نهائي محدث: الأقسام + مدير القسم + موظف الشهر + طلبات الإجازة

## 1) الملخص النهائي المعتمد

### A) الهيكل التنظيمي
- كل موظف يكون له قسم واحد فقط.
- كل قسم له مدير قسم واحد.
- عند تعيين شخص كمدير قسم: كل موظفي هذا القسم يصبحون تحت إشرافه مباشرة.
- يظهر تحت اسم كل شخص في النظام:
  - الوظيفة واسم القسم بين قوسين.
  - إذا كان مدير قسم: يظهر مدير قسم {اسم القسم}.

### B) الأدوار المعتمدة
- Admin و HR و Manager الحالي يظلون بصلاحياتهم الكاملة مثل الآن.
- إضافة Role جديدة مستقلة: department_manager.
- department_manager ليست نفس manager.
- أي موظف عادي يمكن ترقيته إلى department_manager من خلال Admin أو HR أو Manager.
- يمكن أيضا إضافة موظف لقسم بدون ترقيته لمدير قسم.

### C) صلاحيات مدير القسم
- يرى موظفي قسمه فقط.
- يستطيع تقييم الأداء اليومي لموظفي قسمه فقط.
- يستطيع إدارة/متابعة مهام موظفي قسمه فقط.
- يرى إحصائيات أعضاء قسمه.
- لا يستطيع تعديل بيانات الموظف الأساسية.
- لا يستطيع رفع ملفات Excel.
- لا يستطيع التحكم في إعدادات الحضور أونلاين.

### D) موظف الشهر (محدث)
- الفائزون يصبحون 4 بدل 3.
- حد التأهل للفوز يصبح 90 نقطة بدل 85.
- مدير القسم لا يظهر كمرشح في مسابقة أفضل موظف.
- مدير القسم يستطيع التصويت.
- مدير القسم يرى للتصويت أعضاء قسمه فقط.

### E) مدير الشهر (محدث)
- يفوز مدير القسم إذا 3 من الـ 4 الأوائل في نفس الشهر من قسمه.
- يكون الفائز شخصا واحدا فقط.

### F) إعلان النتائج
- يتم إيقاف إعلان النتيجة التلقائي.
- لا تظهر النتائج النهائية للعامة إلا بعد الاعتماد الرسمي من HR/Admin/Manager.

### F) طلبات الإجازة (معتمد)
- نوع إجازة واحد فقط.
- الحساب بالأيام التقويمية.
- يكتب الموظف السبب/النوع في الملاحظات.
- HR فقط يمكنه الموافقة الجزئية على عدد أيام أقل.
- إذا رفض طرف من الأطراف: الطلب يغلق، والموظف يرسل طلبا جديدا.
- إذا لا يوجد مدير قسم أو الموظف ليس مرتبطا بقسم: موافقة HR فقط تكفي.

## 2) تأثير القرارات الجديدة على النظام الحالي

1. يجب توسيع users.role لإضافة department_manager.
2. يجب الحفاظ على manager الحالي كـ Admin-like وعدم تقليل صلاحياته.
3. يلزم فصل منطق مدير القسم عن manager القديم في كل مسارات الصلاحيات.
4. يلزم تحويل الوظائف من Enum ثابت إلى جدول ديناميكي.
5. يلزم تعديل مسارات التصويت لتصبح مرئية حسب القسم لمدير القسم فقط.

## 4) مصفوفة الصلاحيات النهائية

### Admin / HR / Manager
- كامل الصلاحيات الحالية.
- إنشاء/تعديل الأقسام.
- تعيين/إلغاء مدير القسم.
- إضافة موظفين للأقسام.
- ترقية موظف إلى department_manager أو إعادته employee.
- HR يدير الموافقات الجزئية في الإجازات.

### Department Manager
- عرض أعضاء قسمه فقط.
- تقييم الأداء اليومي لأعضاء قسمه فقط.
- إدارة مهام أعضاء قسمه فقط.
- التصويت لأفضل موظف لكن من أعضاء قسمه فقط.
- لا يظهر كمرشح في أفضل موظف.
- لا يملك import/settings/employee-edit الحساسة.

### Employee
- طلب إجازة.
- متابعة الرصيد والأهلية.
- التصويت حسب قواعد دوره.

## 5) قواعد الأعمال المعتمدة

### A) تعيين مدير القسم
1. عند تعيين مدير للقسم، يتم حفظه في departments.manager_employee_id.
2. أي موظف في نفس department_id يصبح تحت إشرافه مباشرة منطقيا.
3. لو تم تغيير المدير، الإشراف ينتقل تلقائيا للمدير الجديد.

### B) عرض الاسم أسفل الشخص
1. مستخدم عادي: {الوظيفة} ({القسم}).
2. مدير قسم: مدير قسم {اسم القسم}.

### C) أفضل موظف
1. عدد الفائزين = 4.
2. Minimum ranking score = 90.
3. مدير القسم لا يدخل مرشحا في المسابقة.
4. مدير القسم يشارك كناخب.
5. مدير القسم يرى مرشحي قسمه فقط في صفحة التصويت.

### D) أفضل مدير
1. يتم فحص الـ Top 4 لنفس الشهر.
2. إذا كان 3 من 4 من نفس القسم، مدير هذا القسم يفوز بلقب مدير الشهر.
3. الفائز النهائي مدير واحد فقط.
4. كسر التعادل: الأعلى في متوسط النقاط النهائية للـ 4 الأوائل من قسمه، ثم الأقل Late minutes إجمالا، ثم الأسبق في تاريخ تعيين مدير القسم.

### E) طلب الإجازة
1. Eligibility:
- يجب وجود employment_start_date.
- يجب إتمام required_work_days_before_leave.
2. المسار:
- إذا الموظف لديه مدير قسم: يلزم موافقة HR + مدير القسم.
- إذا لا يوجد مدير قسم: HR فقط.
3. الرفض:
- أي رفض يغلق الطلب نهائيا.
- الموظف يرسل طلبا جديدا.
4. الموافقة الجزئية:
- HR فقط يحدد approved_days أقل من المطلوب.
5. الحساب:
- الأيام تقويمية.

## 6) Workflow نهائي

### A) Workflow الأقسام والتراتبية
1. Admin/HR/Manager ينشئ قسم.
2. Admin/HR/Manager يختار مدير القسم (أي موظف قابل للترقية).
3. النظام يحول Role إلى department_manager عند الترقية.
4. Admin/HR/Manager يربط الموظفين بالقسم.
5. كل موظف بالقسم يصبح تحت المدير مباشرة.

### B) Workflow تقييم الأداء اليومي
1. مدير القسم يفتح شاشة التقييم اليومي.
2. تظهر له فقط Entries الخاصة بأعضاء قسمه.
3. يقيّم ويحفظ الملاحظات.

### C) Workflow التصويت
1. الموظف العادي: يرى المرشحين المؤهلين حسب القواعد.
2. مدير القسم: يرى المرشحين من قسمه فقط.
3. النظام يستبعد department_manager من قائمة المرشحين نهائيا.
4. اعتماد النتائج: 4 فائزين بشرط 90 نقطة كحد أدنى.
5. لا يتم إعلان النتيجة النهائية إلا بعد الاعتماد اليدوي.

### D) Workflow الإجازات
1. الموظف يدخل صفحة طلب الإجازة.
2. النظام يتحقق من الأهلية والرصيد.
3. إذا غير مؤهل: رسالة منع + الأيام المتبقية.
4. إذا مؤهل: يرسل الطلب.
5. HR ومدير القسم يراجعان (أو HR فقط حسب الحالة).
6. عند اكتمال القبول/الجزئي يتم اعتماد الطلب وتحديث الرصيد.
7. عند الرفض يغلق الطلب.

## 7) خطة التنفيذ المحدثة

### Phase 1: قاعدة البيانات
- إضافة department_manager إلى users.role.
- إضافة departments + روابط employees.department_id.
- إضافة job_titles + تحويل employee للوظيفة الديناميكية.
- إضافة جداول الإجازات والبروفايل والرصيد.

### Phase 2: العلاقات والخدمات
- Models وعلاقات Eloquent.
- Services للترقية/إلغاء ترقية مدير القسم.
- Scope موحد لبيانات القسم المرئية.

### Phase 3: الصلاحيات
- Permission Matrix في Policies واضحة.
- تحديث middleware role حيث يلزم.
- فصل واضح بين manager و department_manager.

### Phase 4: واجهات المستخدم
- إضافة إدارة الأقسام.
- تحديث create/edit الموظف لاختيار القسم والوظيفة الديناميكية.
- تحديث بطاقات/جداول المستخدمين لعرض الوظيفة والقسم والصيغة الخاصة بمدير القسم.

### Phase 5: الأداء/المهام/الحضور
- تقييد شاشات التقييم اليومي والمهام وإحصائيات الفريق بمدير القسم.
- إضافة كروت إحصائية مختصرة لمدير القسم.

### Phase 6: موظف الشهر + مدير الشهر
- رفع عدد الفائزين إلى 4.
- رفع عتبة النقاط إلى 90.
- تصويت مدير القسم داخل قسمه فقط.
- استبعاد مدير القسم من المرشحين.
- إضافة منطق مدير الشهر (3 من 4 في نفس الشهر).

### Phase 7: طلبات الإجازة
- صفحة الموظف (تقديم + أهلية + رصيد + تاريخ).
- صفحة HR/مدير القسم (قائمة + تفاصيل + قرارات + ملاحظات).
- منطق الموافقة الثنائية والجزئية والإغلاق عند الرفض.

### Phase 8: الاختبارات
- Unit tests للمنطق الجديد.
- Feature tests للصلاحيات والفلترة ومسارات الإجازات.
- Regression tests لمسارات manager/admin/hr الحالية.

## 9) المخاطر الفنية المتوقعة

1. إضافة role جديد داخل نظام role الحالي تحتاج مراجعة شاملة للقيود الحالية.
2. التحويل من Enum وظائف إلى جدول ديناميكي يحتاج Migration آمن وBackfill.
3. منطق التصويت المقيد بالقسم لمدير القسم يجب ألا يؤثر على باقي المصوتين.
4. احتساب مدير الشهر يجب أن يكون حتميا وواضحا عند التعادل.

## 10) أسئلة متبقية فقط للحسم قبل بدء الكود

1. عند نقل موظف من قسم لقسم آخر داخل نفس الشهر، في مسابقة مدير الشهر يتم احتسابه لأي قسم:
- القسم الحالي وقت إعلان النتيجة؟
- أم القسم الذي كان فيه وقت اعتماد نتيجة أفضل موظف؟
